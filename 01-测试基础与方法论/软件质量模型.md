# 软件质量模型

## 一句话本质

软件质量模型 = GB/T 25010-2016 定义的八大质量属性（功能性、性能效率、易用性、兼容性、可靠性、安全性、维护性、可移植性），用于系统化定义测试范围、识别质量风险和避免非功能性需求遗漏。

## 核心内容

### 1. 软件质量由八个核心属性构成

根据 **GB/T 25010-2016**，软件质量可划分为：功能性、性能效率、易用性、兼容性、可靠性、安全性、维护性、可移植性。每个属性下又包含若干子特性，例如可靠性包含成熟性、可用性、容错性和可恢复性。

### 2. 不同业务对质量属性的优先级不同

金融系统优先保障可靠性和安全性，电商系统优先保障性能效率和可用性，内部工具可能更看重易用性和可维护性。测试策略应基于业务风险确定质量属性优先级，而非平均用力。

### 3. 每个质量属性都对应明确的测试切入点

| 质量属性 | 典型测试关注点              | 常用测试方法          |
| ---- | -------------------- | --------------- |
| 功能性  | 需求覆盖度、边界条件、异常输入、权限控制 | 等价类、边界值、场景法     |
| 可靠性  | 故障恢复、容错机制、长时间稳定性     | 可靠性测试、混沌工程、故障注入 |
| 性能效率 | 响应时间、吞吐率、资源利用率、容量上限  | 负载测试、压力测试、容量测试  |
| 易用性  | 界面一致性、操作路径、错误提示友好性   | 探索式测试、用户验收测试    |
| 兼容性  | 浏览器/操作系统/版本兼容、数据格式兼容 | 兼容性矩阵测试         |
| 安全性  | 越权访问、数据泄露、注入攻击       | 安全测试、渗透测试       |
| 可维护性 | 日志完整性、监控可观测性、代码可测试性  | 静态分析、可观测性验证     |
| 可移植性 | 环境迁移、配置管理、依赖兼容性      | 环境迁移测试、容器化部署验证  |

### 4. 质量模型应在需求阶段介入

测试遗漏的天花板往往在需求阶段已确定。用质量八要素逐项检查需求，能提前发现缺失的非功能性要求，避免上线后才发现性能、安全或兼容性缺陷。

## 适用场景

- 需求评审阶段，识别需求中的质量要求
- 测试范围定义时，避免遗漏非功能性需求
- 质量风险评估，确定测试优先级
- 非功能性需求评审会议
- 测试计划与测试报告的组织

## 典型反模式

- 只关注功能测试，把性能、安全、兼容性等非功能属性放在最后甚至忽略
- 知道质量八要素，但需求评审时从不对照检查
- 非功能性需求描述模糊（如"系统要快"），就直接进入测试设计
- 所有质量属性同等对待，没有根据业务特点确定优先级
- 把质量模型当纯理论背诵，没有落地到测试策略和用例设计

## 最小可复现实践

1. **建立质量八要素 Checklist**：打印或在文档中列出八大属性，需求评审时逐项勾选是否已明确。
2. **为当前项目定义质量优先级**：用高/中/低给八个属性打分，输出一页质量策略。
3. **用质量维度组织测试计划**：在测试计划中按功能性、性能、可靠性等维度列出测试活动和验收标准。

## 行动建议

- **本周可以尝试**：选一个最近评审的需求文档，用质量八要素逐项检查，记录缺失的质量要求。
- **本月可以输出**：为团队写一份《质量属性优先级评估表》模板。
- **本季度可以落地**：在测试计划评审中强制加入质量模型检查环节。
