# 测试分层策略

> **摘要**：理解测试分层的本质，建立从单元到端到端的分层测试思维，让不同层级的测试各司其职、协同保障质量。

## 适用场景

- 设计自动化测试体系时，需要决策不同测试类型的覆盖范围
- 团队出现测试重复、遗漏或反馈过慢的问题
- 评估项目质量风险，分配测试资源
- 与开发、运维讨论质量门禁和 CI/CD 集成策略

## 关键词

`测试金字塔` `单元测试` `接口测试` `UI 测试` `测试成本`

## 对测试开发工作有什么用？

- 帮助测试开发工程师设计合理的自动化测试架构
- 避免把所有测试都堆在 UI 层，导致执行慢、维护难
- 明确各层测试的职责边界，减少与开发团队的职责冲突
- 为 CI/CD 流水线中的质量门禁提供分层依据

## 测试切入点在哪里？

- **单元测试层**：验证最小代码单元（函数、类、方法）的正确性
- **接口/服务层**：验证模块间契约、业务逻辑、数据流转
- **UI 层**：验证用户交互流程和界面行为
- **端到端层**：验证完整业务链路在真实或准真实环境中的行为
- **探索式/验收层**：发现预设用例覆盖不到的意外风险

## 核心内容

### 1. 测试金字塔

测试金字塔强调底层测试（单元/接口）占比高、执行快、成本低，顶层测试（UI/端到端）占比低、覆盖关键路径。

```text
    /\   端到端 / 探索式测试  ← 少而精
   /  \  UI 自动化测试
  /____\ 接口 / 服务层测试
 /______\ 单元测试          ← 多而快
```

### 2. 各层测试的定位

| 层级 | 关注点 | 执行速度 | 维护成本 | 典型工具 |
|---|---|---|---|---|
| 单元测试 | 函数/类正确性 | 快 | 低 | Pytest、JUnit、Mock |
| 接口测试 | 契约、业务逻辑 | 较快 | 中 | Pytest + requests、Postman |
| UI 测试 | 用户交互、页面行为 | 慢 | 高 | Playwright、Selenium |
| 端到端测试 | 完整业务流程 | 最慢 | 最高 | 自动化脚本 + 真实环境 |

### 3. 分层的决策依据

- **成本与收益**：底层测试投资回报高，应优先覆盖
- **反馈速度**：越靠近开发阶段，反馈越快
- **稳定性**：底层测试受环境变化影响小
- **业务风险**：核心链路需要端到端验证作为最终保障

### 4. 常见反模式

- **冰淇淋模型**：大量 UI 测试、少量单元测试，导致执行慢、不稳定
- **测试重复**：同一逻辑在多层重复验证
- **边界模糊**：单元测试做了集成测试的事，或接口测试验证了 UI 细节
