# 测试分层策略

## 一句话本质

测试分层策略 = 按单元 → 接口 → UI → 端到端的层级分配测试职责，用金字塔形状控制成本与反馈速度，让不同层级各司其职、互为补充。

## 核心内容

### 1. 测试金字塔是分层的基本原则

测试金字塔强调底层测试（单元/接口）占比高、执行快、成本低，顶层测试（UI/端到端）占比低、覆盖关键路径。

```text
    /\   端到端 / 探索式测试  ← 少而精
   /  \  UI 自动化测试
  /____\ 接口 / 服务层测试
 /______\ 单元测试          ← 多而快
```

### 2. 各层测试有不同定位

| 层级    | 关注点       | 执行速度 | 维护成本 | 典型工具                      |
| ----- | --------- | ---- | ---- | ------------------------- |
| 单元测试  | 函数/类正确性   | 快    | 低    | Pytest、JUnit、Mock         |
| 接口测试  | 契约、业务逻辑   | 较快   | 中    | Pytest + requests、Postman |
| UI 测试 | 用户交互、页面行为 | 慢    | 高    | Playwright、Selenium       |
| 端到端测试 | 完整业务流程    | 最慢   | 最高   | 自动化脚本 + 真实环境              |

### 3. 分层决策依据

- **成本与收益**：底层测试投资回报高，应优先覆盖
- **反馈速度**：越靠近开发阶段，反馈越快
- **稳定性**：底层测试受环境变化影响小
- **业务风险**：核心链路需要端到端验证作为最终保障

### 4. 常见分层反模式

- **冰淇淋模型**：大量 UI 测试、少量单元测试，导致执行慢、不稳定
- **测试重复**：同一逻辑在多层重复验证
- **边界模糊**：单元测试做了集成测试的事，或接口测试验证了 UI 细节

## 适用场景

- 设计自动化测试体系时，需要决策不同测试类型的覆盖范围
- 团队出现测试重复、遗漏或反馈过慢的问题
- 评估项目质量风险，分配测试资源
- 与开发、运维讨论质量门禁和 CI/CD 集成策略

## 典型反模式

- 冰淇淋模型：UI 测试占比过高，底层测试不足
- 同一逻辑在单元、接口、UI 多层重复验证
- 各层职责边界模糊，单元测试测集成，接口测试验 UI
- 为了分层而分层，忽视业务实际风险和团队能力
- 完全依赖自动化分层，忽视探索式测试和验收测试的补充价值

## 最小可复现实践

1. **画出当前项目的测试金字塔**：统计单元、接口、UI、端到端用例数量与执行时间。
2. **定义每层职责边界**：明确什么类型的验证应该放在哪一层，写入团队规范。
3. **为每层设定目标**：例如单元测试覆盖率 ≥ 70%、接口测试执行时间 < 5 分钟、UI 测试只覆盖核心链路。

## 行动建议

- **本周可以尝试**：盘点当前项目各层用例占比，画出实际的测试金字塔形状。
- **本月可以输出**：一份《分层测试策略》文档，明确每层职责、工具和门禁标准。
- **本季度可以落地**：在 CI/CD 流水线中建立分层执行机制，例如提交时跑单元+接口，合并前跑 UI+端到端。
