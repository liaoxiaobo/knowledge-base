# 测试分析与设计

## 一句话本质

测试分析解决"测什么"（从需求与风险中识别测试条件），测试设计解决"怎么测"（选择黑盒、白盒或经验方法生成用例），两者共同构成系统化、可度量的测试覆盖。

## 核心内容

### 1. 测试分析与设计是两个前后衔接的活动

- **测试分析（Test Analysis）**：依据测试依据（需求规格、用户故事、风险列表、代码结构等）识别需要测试的内容，输出**测试条件（Test Conditions）**。例如"验证登录密码强度校验"就是一个测试条件。
- **测试设计（Test Design）**：将测试条件具体化为**测试用例**和**测试数据**，并选择适当的设计技术。
- **常见流程**：测试条件 → 测试用例 → 测试数据 → 测试规程。

### 2. ISTQB 将测试设计技术分为三大类

| 类别         | 别称   | 关注焦点       | 典型方法                 |
| ---------- | ---- | ---------- | -------------------- |
| **基于规格说明** | 黑盒测试 | 输入与输出、需求行为 | 等价类、边界值、决策表、状态迁移、场景法 |
| **基于结构**   | 白盒测试 | 内部代码结构与逻辑  | 语句覆盖、判定覆盖、条件覆盖、路径覆盖  |
| **基于经验**   | 经验方法 | 测试人员知识与直觉  | 错误推测、探索式测试、基于会话的测试   |

### 3. 黑盒方法覆盖需求行为

| 方法             | 核心逻辑                     | 适用场景              |
| -------------- | ------------------------ | ----------------- |
| **等价类划分**      | 将输入划分为有效/无效等价类，每类选一个代表   | 输入条件较多，需减少用例数     |
| **边界值分析**      | 测试等价类边界值，如最小值、最大值、临界值    | 数值范围、长度限制、分页大小    |
| **决策表测试**      | 把条件组合与对应动作整理成表格，每列生成一个用例 | 多条件组合、业务规则复杂      |
| **状态迁移测试**     | 识别系统状态和触发事件，覆盖有效/无效迁移    | 工作流、订单状态、审批流程     |
| **场景法 / 用例测试** | 基于用户操作流程，覆盖基本流和备选流       | 端到端业务流程、验收测试      |
| **分类树**        | 将输入维度拆分为独立分类，再组合生成用例     | 多维度参数组合           |
| **错误推测法**      | 基于经验测试易错点，以点带面挖掘共性 bug   | 特殊字符、越权、并发、历史高发缺陷 |

### 4. 白盒方法覆盖代码结构

| 方法          | 定义                  | 覆盖强度     |
| ----------- | ------------------- | -------- |
| **语句覆盖**    | 每条可执行语句至少执行一次       | 最低       |
| **判定覆盖**    | 每个判定（分支）的真/假至少执行一次  | 基础       |
| **条件覆盖**    | 每个条件表达式的真/假至少执行一次   | 中等       |
| **判定-条件覆盖** | 同时满足判定覆盖和条件覆盖       | 较高       |
| **条件组合覆盖**  | 每个判定中所有条件取值组合至少执行一次 | 高        |
| **路径覆盖**    | 覆盖所有可能的执行路径         | 最高，成本也最高 |

### 5. 经验方法补充规范化设计的盲区

- **错误推测法**：不依赖系统方法，而是基于测试人员对类似系统、历史缺陷的直觉快速定位高风险点。
- **探索式测试**：测试设计与执行同步进行，适合新功能快速验证、补充自动化难以覆盖的认知盲区。
- **基于会话的测试管理（SBTM）**：为探索式测试设定时间盒（如 60–90 分钟）和测试章程，提高可追溯性。

### 6. 方法选择应基于风险与成本

- **先黑盒后白盒**：先用黑盒方法覆盖需求，再用白盒覆盖率数据找到未覆盖的代码路径，补充针对性用例。
- **高风险优先**：核心链路、历史缺陷集中区域优先使用场景法 + 错误推测法 + 判定-条件覆盖。
- **避免过度覆盖**：路径覆盖和条件组合覆盖成本极高，通常只在安全关键或核心算法模块使用。
- **组合使用**：没有一种方法能发现所有缺陷，实际项目需要多种方法互补。

## 适用场景

- 测试条件识别与测试用例设计
- 测试方案制定与评审
- 自动化测试脚本覆盖设计
- 白盒覆盖率提升
- 面试准备与技术分享

## 典型反模式

- 混淆测试分析与测试设计，跳过测试条件直接写用例步骤
- 只用黑盒或只用白盒，忽视两者互补
- 用例设计脱离真实用户场景，变成孤立的功能点验证
- 忽视边界值、异常流和无效等价类，只测 happy path
- 所有功能同等用力，没有基于风险优先级分配测试资源
- 用例集长期不更新，陷入杀虫剂悖论
- 追求 100% 路径覆盖，导致维护成本失控
- 把探索式测试当作"随便点点"，没有测试章程和时间盒

## 最小可复现实践

1. **用思维三问识别测试条件**：这个需求要验证什么？哪些输入/状态组合有风险？历史类似功能哪里容易出错？
2. **为每个测试条件选择设计技术**：输入密集型用等价类 + 边界值；多条件组合用决策表；有状态变化用状态迁移。
3. **用场景法串联用例**：把孤立的单点用例组织成"用户从 A 操作到 B，再触发 C 异常"的端到端流程。
4. **用白盒覆盖率查漏补缺**：运行测试后查看语句/判定覆盖，对未覆盖路径补充用例。

## 行动建议

- **本周可以尝试**：选一个功能，先列出 5–10 个测试条件，再用等价类 + 边界值设计一组用例。
- **本月可以输出**：为团队写一份《用例设计方法选择指南》或《测试条件识别 Checklist》。
- **本季度可以落地**：在一个核心模块中建立"黑盒需求覆盖 + 白盒覆盖率门禁"的组合策略。
