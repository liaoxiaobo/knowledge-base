# 测试左移与右移

## 一句话本质

测试左移与右移 = 把质量意识和测试能力前置到需求、设计、编码阶段（左移），并在生产环境持续验证和反馈（右移），形成覆盖软件全生命周期的质量闭环。

## 核心内容

### 1. 测试左移不是简单提前测试，而是注入质量意识

测试左移的核心是把测试能力注入软件生命周期的上游，在缺陷注入之前就进行预防。

| 阶段  | 左移实践           | 测试开发角色          |
| --- | -------------- | --------------- |
| 需求  | 需求可测试性评审、实例化需求 | 提出边界场景、设计验收标准   |
| 设计  | 接口契约评审、异常场景设计  | 设计接口测试策略        |
| 编码  | 代码评审、静态分析、单元测试 | 提供测试框架、辅助编写单元测试 |
| 提交  | 预提交检查、增量测试     | 搭建本地/CI 快速反馈流水线 |

### 2. 测试右移强调上线后的持续验证

测试右移认为软件发布不是质量保障的终点，生产环境仍需持续反馈。

| 阶段  | 右移实践              | 测试开发角色        |
| --- | ----------------- | ------------- |
| 发布  | 灰度发布、A/B 测试、金丝雀验证 | 设计发布验证用例      |
| 生产  | 监控告警、日志分析、性能基线    | 构建可观测性测试、异常检测 |
| 故障  | 复盘、混沌工程、回归加固      | 沉淀故障场景为自动化用例  |

### 3. 左移与右移互为补充

- **左移**降低缺陷注入概率，让问题在成本最低的阶段被发现。
- **右移**降低缺陷影响半径，验证真实环境下的系统行为。
- 两者结合形成"预防 → 发现 → 恢复 → 加固"的完整质量反馈闭环。

### 4. 实践中的常见挑战

- 左移需要测试具备需求理解和代码阅读能力。
- 右移需要测试与运维、SRE 深度协作。
- 过度左移可能导致测试早期介入但缺乏有效输出，反而增加沟通成本。
- 右移需要平衡监控成本与收益，避免告警疲劳。

## 适用场景

- 团队希望更早发现缺陷，减少后期返工
- 生产环境问题频发，需要建立发布后的质量反馈机制
- 推进 DevOps 或持续交付，需要测试与开发、运维深度协作
- 设计 CI/CD 流水线时，确定测试在不同阶段的介入点

## 典型反模式

- 左移变成测试"提前背锅"，只增加评审负担没有实质输出
- 右移变成只加监控不行动，告警出来无人响应
- 左右移都流于形式，没有可落地的检查项和责任人
- 过度左移，在需求极不稳定阶段就投入大量测试设计，导致频繁返工
- 右移缺乏与运维协作，测试无法获取生产数据也不敢做发布验证

## 最小可复现实践

1. **建立需求可测试性评审 Checklist**：在需求评审时检查验收标准是否包含功能和非功能维度。
2. **设计发布验证用例模板**：明确灰度/金丝雀阶段需要验证哪些核心链路。
3. **建立生产故障复盘模板**：记录现象、根因、修复、沉淀为自动化用例或监控项。

## 行动建议

- **本周可以尝试**：参与一次需求评审，用可测试性 Checklist 记录 3 个可改进点。
- **本月可以输出**：一份《发布验证用例集》或《生产故障复盘模板》。
- **本季度可以落地**：引入一个右移实践，例如核心接口生产监控、灰度发布验证或混沌工程演练。
