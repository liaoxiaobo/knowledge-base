# 软件质量模型与标准

> **摘要**：理解 GB/T 25010-2016 软件质量八要素，建立从功能性到安全性的全质量属性视角，支撑测试范围定义和风险评估。

## 适用场景

- 测试需求分析阶段，识别需求中的质量要求
- 测试范围定义时，避免遗漏非功能性需求
- 质量风险评估，确定测试优先级
- 非功能性需求评审会议

## 关键词

`GB/T 25010-2016` `软件质量八要素` `非功能性测试` `质量风险`

## 对测试开发工作有什么用？

- 帮助测试开发工程师跳出"只测功能"的局限，系统化覆盖质量属性
- 为自动化测试和非功能性测试提供理论依据
- 在需求评审阶段识别缺失的质量要求
- 指导测试计划、测试策略和测试报告的组织方式

## 测试切入点在哪里？

- **需求阶段**：用质量八要素逐项检查需求完整性
- **测试设计**：为每个质量属性设计对应测试类型
- **风险评估**：根据业务特点确定质量属性优先级
- **测试报告**：用质量维度组织测试结果和覆盖率

## 核心内容

### 1. 软件质量八要素

从管理角度对软件质量进行度量，根据 **GB/T 25010-2016**，影响软件质量的主要因素可划分为 8 个属性：功能性、性能效率、易用性、兼容性、可靠性、安全性、维护性与可移植性。

| 特性       | 核心子特性                                       |
| -------- | ------------------------------------------- |
| **功能性**  | 功能完备性、功能适合性、功能正确性                           |
| **可靠性**  | 成熟性（无故障）、可用性、容错性（出了故障不影响主功能）、可恢复性（快速恢复）     |
| **易用性**  | 易理解性、易学性、易操作性                               |
| **性能效率** | 时间特性（RT≤2s）、资源利用性（CPU≤80%）                  |
| **可维护性** | 模块化、易分析性、易修改性、易测试性                          |
| **可移植性** | 适应性、易安装性、易替换性                               |
| **兼容性**  | 共存性（与其他子系统兼容、版本数据兼容、平台硬件兼容）、互操作性（接口标准化、开放性） |
| **安全性**  | 机密性、完整性、可审计性                                |

### 2. 测试视角：每个质量属性的测试切入点

| 质量属性 | 典型测试关注点              | 常用测试方法          |
| ---- | -------------------- | --------------- |
| 功能性  | 需求覆盖度、边界条件、异常输入、权限控制 | 等价类、边界值、场景法、因果图 |
| 可靠性  | 故障恢复、容错机制、长时间稳定性     | 可靠性测试、混沌工程、故障注入 |
| 性能效率 | 响应时间、吞吐率、资源利用率、容量上限  | 负载测试、压力测试、容量测试  |
| 易用性  | 界面一致性、操作路径、错误提示友好性   | 探索式测试、用户验收测试    |
| 兼容性  | 浏览器/操作系统/版本兼容、数据格式兼容 | 兼容性矩阵测试         |
| 安全性  | 越权访问、数据泄露、注入攻击       | 安全测试、渗透测试       |
| 可维护性 | 日志完整性、监控可观测性、代码可测试性  | 静态分析、可观测性验证     |
| 可移植性 | 环境迁移、配置管理、依赖兼容性      | 环境迁移测试、容器化部署验证  |

### 3. 实践建议

1. **需求评审时使用质量模型**：对照八大属性逐项检查需求是否明确了质量要求，避免遗漏非功能性需求。
2. **测试计划阶段识别质量风险**：根据业务特点确定优先级，例如金融系统优先保障可靠性和安全性，电商系统优先保障性能和可用性。
3. **设计测试策略时覆盖全属性**：不要只关注功能性，性能、安全、兼容性往往是线上问题高发区。
