# 测试流程与敏捷实践

## 一句话本质

测试流程与敏捷实践 = 根据项目节奏选择瀑布或敏捷测试模式，通过测试左移、持续测试和探索式测试，把质量反馈嵌入软件交付全生命周期。

## 核心内容

### 1. 传统测试流程是阶段式推进

1. **需求分析**：明确测试范围与优先级。
2. **测试计划**：制定目标、策略、风险应对方案。
3. **测试设计**：转化为具体用例（等价类、场景法等）。
4. **测试执行**：执行自测、回归测试、缺陷跟踪。
5. **发布维护**：监控线上问题并修复。

传统流程适合需求稳定、发布周期长的项目，但缺陷发现晚、修复成本高。

### 2. 敏捷测试强调短、频、快的质量反馈

敏捷测试本质是持续反馈代码提交的质量，促进持续交付。

![敏捷测试流程](../assets/1.3-agile-testing-flow.png)

#### 2.1 敏捷测试特点

- **质量内建**：测试左移（参与需求评审） + 测试右移（监控生产环境）。
- **持续测试**：自动化与探索式测试结合，快速反馈代码质量。
- **分层策略**：单元测试 → 接口测试 → UI 测试，平衡效率与覆盖。

#### 2.2 敏捷测试中的测试角色

| 阶段  | 测试活动               |
| --- | ------------------ |
| 需求  | 参与评审、识别模糊点、定义验收标准  |
| 开发  | 代码评审、单元测试评审、接口契约验证 |
| 集成  | 接口自动化、持续集成验证       |
| 发布  | 回归测试、发布准出          |
| 运维  | 线上监控、故障复盘、A/B 测试   |

### 3. 探索式测试补充自动化盲区

探索式测试将学习、测试设计、测试执行和结果分析作为一个循环快速迭代，在较短时间内完成多次循环，持续优化测试。

- **核心**：质疑系统存在漏洞（需求误解、实现错误、性能瓶颈等）。
- **实施步骤**：
  1. 制定 SMART 目标（具体、可度量、可实现）。
  2. 分时间盒（如 50 分钟）执行"设计-执行-分析"循环。
  3. 通过测程管理（SBTM）记录测试结果并复盘。
- **适用场景**：新功能快速验证、复杂业务流程首次测试、自动化覆盖不足的长尾场景、发现"意料之外"的问题。

### 4. 流程选择应匹配项目特征

- **需求稳定、周期长**：传统瀑布仍有价值，注意在测试设计阶段充分覆盖非功能性需求。
- **需求变化快、迭代频繁**：敏捷测试更合适，测试人员应尽早介入需求评审。
- **无论哪种模式**：核心链路自动化保障回归效率，探索式测试补充长尾和异常场景。

## 适用场景

- 团队流程设计
- 敏捷转型
- 测试计划制定
- CI/CD 流水线规划

## 典型反模式

- 敏捷转型中测试仍是"最后守门员"，没有参与需求评审和代码评审
- 用自动化测试完全替代探索式测试，忽视人的认知价值
- 只关注开发阶段测试，忽视生产环境监控和右移反馈
- 流程照搬，不根据团队成熟度和业务特征调整
- 探索式测试没有目标和时间盒，变成"随便点点"

## 最小可复现实践

1. **在需求评审中加入验收标准定义**：用 Given-When-Then 或验收条件清单明确可测性。
2. **建立持续集成验证环节**：每次代码提交自动触发单元测试和接口测试，快速反馈。
3. **每周安排一次时间盒化的探索式测试**：用 SBTM 记录测试章程、时间和发现。

## 行动建议

- **本周可以尝试**：观察一次当前迭代的需求评审，记录测试人员介入的时机和可改进点。
- **本月可以输出**：为团队建立一份《探索式测试 SBTM 记录模板》。
- **本季度可以落地**：将核心链路自动化测试接入 CI，实现每次提交自动验证。
