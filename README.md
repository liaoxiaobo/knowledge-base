# 测试开发知识库

个人测试开发知识库。将博客存量文章、项目实战经验、AI 测试能力沉淀为一套分类清晰、可检索、可复用、可持续积累的专业资产，同时服务于系统性学习和简历输出。

主要入口：
- 内容地图与学习路径：[MOC-首页.md](MOC-首页.md)
- 仓库规范与模板选择：见下文

---

## 目录结构

```
knowledge-base/
├── 00-inbox/                  # 临时收集箱：碎片想法、待整理素材、剪贴内容
├── 01-测试基础与方法论/        # 道：质量观、测试思维、流程方法
├── 02-测试开发基本技能/        # 器：Python、Linux、网络、数据库、Git
├── 03-自动化测试工程/          # 柱：框架、UI/接口自动化、CI/CD
├── 04-云计算领域测试/          # 柱：K8s、云产品、中间件、环境管理
├── 05-专项测试技术/            # 扩：性能、稳定性、混沌工程、安全
├── 06-质量效能平台与数据/      # 数：报告、度量、失败归因、资产
├── 07-团队组织力与影响力/      # 人：流程规范、缺陷管理、质量文化
├── 08-AI与智能化测试/          # 柱：AI 编程、AI 辅助测试、LLM/RAG 测试
├── _archive/                # 归档区
├── _templates/                # 文档模板库
├── assets/                    # 图片、截图、PDF 等附件
├── MOC-首页.md                # 知识库总入口与学习地图
└── 知识库设计方案.md           # 设计原则与实施路线图
```

## 核心设计原则

- **按领域聚合，不按来源拆分**：通用知识和项目经验放在同一目录下，通过标签区分类型
- **AI 能力独立成域，同时保持领域关联**：AI 内容集中到 `08-AI与智能化测试`，通过 MOC 关联到 01-07 各域
- **知识库服务职业发展**：每篇项目文档都应能回答简历四问（做了什么、解决什么问题、用了什么技术、取得什么量化结果）
- **可持续沉淀**：通过命名规范、标签体系、模板和定期 Review 机制，保证知识库长期有效

## 命名规范

- **目录名**：`[编号]-主题名`，编号用于排序和学习路径标识
  - 示例：`01-测试基础与方法论`、`03-自动化测试工程`
- **文件名**：`主题名.md`，不嵌入编号，保持简洁
  - 示例：`软件质量模型.md`、`Pytest.md`、`K8s 部署.md`
- **页面标题**：与文件名保持一致，不嵌入编号
- **图片命名**：`领域-主题-序号.png`

## 笔记属性

每篇文档顶部使用 YAML frontmatter 描述元数据。`tags` 是其中最重要的字段，用于表达文档类型、技术栈和场景。

### 常用字段

| 字段        | 必填     | 说明                     | 示例                        |
| --------- | ------ | ---------------------- | ------------------------- |
| `title`   | 是      | 文档标题，与文件名保持一致          | `测试工程师的核心竞争力`             |
| `tags`    | 是      | 文档类型 + 技术栈 + 场景标签      | `["理论", "规范", "测试工程师成长"]` |
| `created` | 是      | 创建日期，格式 `YYYY-MM-DD`   | `2026-06-22`              |
| `updated` | 是      | 最近更新日期，格式 `YYYY-MM-DD` | `2026-07-02`              |
| `domain`  | 建议     | 所属一级领域，便于跨域检索          | `07-团队组织力与影响力`            |
| `project` | 项目文档必填 | 具体项目名称                 | `CCE 容器监控`                |


### 文档类型

| 标签    | 含义             | 示例           |
| ----- | -------------- | ------------ |
| `#理论` | 概念、原理、方法论      | 软件质量模型       |
| `#实战` | 通用技术实践         | k8s 部署       |
| `#项目` | 来自具体业务项目       | CCE 测试实践     |
| `#复盘` | 问题排查、故障复盘      | 节点调度异常根因分析   |
| `#规范` | 团队规范、Checklist | 用例设计规范       |
| `#速查` | 命令、配置、快捷参考     | kubectl 常用命令 |

### 技术栈与场景

`Python` `Pytest` `Playwright` `Selenium` `RobotFramework` `Jenkins` `Docker` `Kubernetes` `Nginx` `Git` `ChaosMesh` `ChaosBlade` `Locust` `UI自动化` `接口测试` `CI/CD` `混沌工程` `性能测试` `环境搭建` `失败归因` `AI测试` `LLM` `RAG`

## 模板选择指南

模板位于 `_templates/` 目录。新建文档时，根据内容类型选择对应模板：

| 模板                                 | 适用场景                  | 常见反模式                  |
| ---------------------------------- | --------------------- | ---------------------- |
| [通用知识笔记模板](_templates/通用知识笔记模板.md) | 方法论、原则、最佳实践、能力模型、复盘经验 | 把速查命令写成理论文章；把项目细节混进通用知识；用通用模板写面试题 |
| [项目实战文档模板](_templates/项目实战文档模板.md) | 具体业务项目的测试实践、简历素材沉淀    | 只列技术栈不写业务价值；不写简历四问；把通用方法论当项目经验   |
| [速查表模板](_templates/速查表模板.md)       | 命令、配置、参数、快捷参考         | 在速查表里展开长篇原理；只写标题没有具体命令/参数；缺少常见问题   |
| [面试题模板](_templates/面试题模板.md)       | 面试题整理、考察点拆解、参考答案      | 把非面试知识硬套成面试题；只给答案不写思路；缺少延伸问题和关联知识点 |

## 写作流程

1. **收集**：碎片内容先扔进 `00-inbox/`，不纠结分类
2. **整理**：定期清理 inbox，有价值的内容使用 `_templates/` 中的模板重写
3. **归档**：无价值的内容删除或移入 `_archive/`
4. **链接**：在文档中用 `[[文件名]]` 建立双向链接
5. **更新**：新增或调整领域文档后，回到 `MOC-首页.md` 更新入口和状态

## 工具配置

本知识库主要配合 Obsidian 使用：

- **附件目录**：`assets/`（`.obsidian/app.json` 中配置 `attachmentFolderPath: assets`）
- **模板目录**：`_templates/`（`.obsidian/app.json` 中配置 `templates-folder: _templates`）
- **智能关联**：使用 Smart Connections 插件基于向量检索建立笔记关联
