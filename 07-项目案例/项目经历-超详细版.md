# 项目经历

## 项目一：云平台自动化测试体系建设

**项目周期**：2022.06 – 至今

### 项目背景与目标

曙光云（SugonCloud）全栈云平台涵盖计算、存储、网络、数据库、中间件、容器、安全合规等 10 大业务域、50+ 云服务。业务迭代频繁，手工回归测试周期长、覆盖率低，且缺乏对底层虚拟化/存储/网络状态的深度验证能力。项目从零建设覆盖 API + UI 双层的企业级自动化测试体系，实现核心业务流程的全链路自动化回归，支撑敏捷迭代中的质量兜底。

### 我的职责与角色

作为项目负责人兼架构师，全程主导技术方案设计、框架搭建、核心模块开发、CI/CD 流水线建设及团队规范制定，带领 3 人测试开发小组完成体系建设与持续演进。

### 技术架构与选型

**API 自动化层（一期，pytest-sugon）**

- **技术栈**：Python 3 + Pytest 8.2 + Requests + Allure + PyMySQL + Paramiko SSH + Jenkins + Docker
- **架构模式**：API Object 模式 + Tenant 单例聚合。Tenant 类通过多重继承整合 IAM / Compute / Storage / Network / CCE / PaaS 六大模块 API，初始化自动完成登录认证 → 可用区获取 → VPC 初始化 → 镜像查找等全链路准备，一次初始化即获得所有后端能力
- **Fixture 管理**：30+ 个 function 级 fixture 统一遵循"创建 → yield → 自动清理"生命周期，session 级共享 conf / tenant / jump_host 等长生命周期资源
- **验证层次**：API 响应断言 + SSH 底层验证（虚拟化层 VM 物理分布、云盘内核级设备存在性）+ MySQL 数据库直连验证

**UI 自动化层（二期，playwright-sugon）**

- **技术栈**：Python 3 + Playwright 1.53（同步 API）+ Pytest 8.2 + Allure + Jenkins + Docker + 飞书通知
- **分层架构**：严格四层分离——Page Object 层（UI 元素封装）→ Fixture 层（资源生命周期管理）→ Helper 层（业务编排纯函数）→ Test Class 层（用例步骤+断言）
- **BasePage 组合架构**：通过 9 个 Mixin 多重继承构建 BasePage（Navigation / Buttons / Dialogs / Inputs / Selectors / Tables / Actions / Waits / Playwright），业务页面对象再通过 10+ 子 Mixin 二次聚合
- **断言体系独立化**：将断言层从页面对象解耦，按业务域分为 base / compute / network / storage / security / database 等独立包
- **后端验证层**：SSH 类组合 ScliMixin（CLI 命令解析）+ CloudOpsMixin（连通性检测 / 磁盘挂载 / 数据库直连），实现"UI 操作 + 后端验证"闭环

### 核心功能实现

**API 层核心能力**

1. **全栈 API 覆盖**：357 条 API 测试用例覆盖 ECS（64 条）、VPC/SG/ACL/QoS/DNS（91 条）、EVS（21 条）、数据库 4 种引擎（MySQL/PostgreSQL/MongoDB/KingbaseES，71 条）、中间件 4 种（Redis/Kafka/RabbitMQ/ES/Prometheus，33 条）、CCE 容器（20 条）、SFS 文件存储等
2. **后端深度验证**：反亲和性策略通过 SSH 登录计算节点执行 `gova list` 在虚拟化层验证 VM 物理分布；云盘挂载通过 SSH 在内核层验证设备存在性——这些都是 UI 层面无法覆盖的底层验证
3. **Allure 报告自动增强**：`@allure_data` 装饰器自动拦截所有 API 请求/响应，将请求体、响应体、状态码附加到报告，失败时提供完整请求上下文
4. **多存储后端支持**：通过 `--stor` 参数和 `skip_stor` 装饰器，支持 xstor / zbs / ceph / xbd 等多种存储后端灵活切换

**UI 层核心能力**

1. **多角色测试能力**：设计"用户角色透明切换"机制——session 级 `admin_browser_context` 复用管理员登录态，非 admin 角色通过 `ops_page` fixture 自动"借用"admin 能力完成基础设施操作，一套用例兼容 admin / dept_admin / user 三种角色。改造覆盖 IAM 模块 75 个用例、compute/network/backup 等模块 38+ 个用例
2. **多环境并行调度**：Jenkins Pipeline 支持 `ENV_DISPATCH` 参数，将不同模块按 JSON 配置分发到不同环境并行执行，BMS 用例自动强制单进程顺序执行
3. **VM Fixture 依赖注入引擎**：独立子模块实现参数解析 → 依赖解析 → 请求构建 → 资源创建 → 元数据采集 → MFIP 绑定 → 清理管理的完整流水线，支持单批次、多批次不同配置 VM 的自动创建
4. **三层资源清理防护**：Fixture yield finally + cleanup LIFO 兜底 + vm_names 即时注册，确保云资源零泄漏
5. **自动 Mark 注册体系**：自动扫描测试文件动态注册三级 pytest mark，配合最长前缀匹配算法支持灵活筛选

### 遇到的挑战与解决方案

| 挑战 | 解决方案 |
|------|----------|
| 非 admin 用户无权访问 `/ops` 路径的基础设施服务 | 设计 `admin_browser_context` + `ops_page` 透明切换机制，普通用户"借用"admin 能力 |
| Playwright 定位器需大量显式等待代码 | 封装 `CustomLocator`，click/fill 自动触发操作完成等待，减少 60%+ 等待代码 |
| 反亲和验证仅靠 API 返回值不可信 | SSH 登录计算节点执行虚拟化层命令做底层交叉验证 |
| 多环境并行时 Allure result.json 重复合并 | Jenkinsfile 侧按子目录去重，保持 collector 职责单一 |
| 密码明文传输风险 | RSA 公钥加密用户密码再传输 |

### 项目成果与效益

- 建立覆盖 **10 个业务域、50+ 云服务、750+ 条自动化用例**（API 357 条 + UI 400+ 条）的双层测试体系
- API 层平均执行 15 分钟提供快速反馈，UI 层 4-6 小时完成全量回归，人工回归从 3-5 天压缩至自动化
- 核心业务流程自动化覆盖率从 0 提升至约 70%
- 多角色测试能力减少约 60% 的重复用例编写
- 累计拦截线上缺陷 80+ 个，支撑 20+ 次版本迭代

## 项目二：AI 驱动的智能测试体系落地

**项目周期**：2023.03 – 至今

### 项目背景与目标

随着自动化用例规模增长至 400+，CI 流水线失败排查耗时（单次 30-60 分钟），且环境抖动的"假失败"与真实缺陷难以快速区分；同时新需求的测试脚本编写效率低、人工评审遗漏多。项目将 AI/LLM 能力深度集成到测试全流程，覆盖"脚本生成 → 质量门禁 → 失败分析"三大环节，实现测试全链路的 AI 提效赋能。

### 我的职责与角色

作为项目技术负责人，主导 AI 失败分析系统的架构设计、AI 测试脚本生成的硬闸门体系设计、知识库体系搭建，以及 LLM 调用策略优化。

### 技术架构与选型

- **AI 提供商**：DeepSeek / DashScope 双 LLM 提供商，支持灵活切换和成本优化
- **失败分析链**：规则分类器 → 相似聚合器 → LLM 根因推断 → 结构化报告，采用"规则优先、LLM 兜底"策略
- **脚本生成闸门链**：precheck（静态门禁）→ pytest 执行 → loop_gate（循环控制）→ report_check（报告校验），退出码驱动的硬控制体系
- **证据等级体系**：Playwright Trace > 截图 > 日志 > 代码 > 推断

### 核心功能实现

**一、AI 失败自动分析**

1. **智能分类与聚合**：规则分类器通过关键词匹配（系统升级中/503/SSH 不通/TimeoutError/AssertionError）快速分为 ENVIRONMENT / TEST_BUG / PRODUCT_BUG / UNKNOWN 四类。相似聚合器按 `(分类, 堆栈帧, 错误消息)` 三元组合并同类失败，同类只调一次 LLM
2. **LLM 根因推断**：通过提示词工程注入用例源码片段、截图/trace 证据、历史案例库上下文，产出结构化分析报告（根因分类 + 置信度 + 证据引用 + 修复建议），生成 `ai-test-summary.md` + `failure-report.json` 并归档
3. **知识库闭环**：沉淀典型失败模式案例库（页面加载超时、资源异步创建未就绪、浮层遮挡点击失败等），命中案例库直接匹配不走 LLM；工程师深度分析结论回写案例库反哺自动化层

**二、AI 测试脚本自动生成与质量保障**

设计了一套嵌入 AI 生成流程的**多层硬闸门（Hard Gate）体系**，将质量约束从软 prompt 升级为退出码驱动的机械判定：

1. **静态门禁 precheck**（零执行成本前置拦截）：
   - 7 类硬违规检测：禁止测试层直接使用底层 API（`.locator()`、`xpath=`）、禁止固定等待（`wait_for_timeout`、`time.sleep`）、禁止测试方法体 try/except 包裹、禁止 SSH 命令掩盖错误（`|| true`）、新增 marker 必须登记、参数化用例数必须等于需求场景数、禁止 JS 填表反模式（`dispatchEvent` 合成事件等）
   - 使用 AST 精确判定（非正则）、conftest 链递归解析 Page Object 导入、`git show HEAD` 检测已有用例是否被误删
   - Recon 门禁：新建 Page Object 前必须有运行时侦察产物，防止 AI "凭空猜页面结构"

2. **循环闸门 loop_gate**：读取运行报告中的结构化计数区（全局修复轮次 / 单用例修复轮次 / 状态冻结检测 / 同根因修复次数），五项停止条件机械判定是否继续回退修复，支持全局端到端总时长闸兜底

3. **报告校验 report_check**：校验各阶段运行报告的锚点完整性（共 6 个阶段 27 个必出锚点），强制完成时间精确到秒，防止 AI 凭记忆遗漏关键区块

### 遇到的挑战与解决方案

| 挑战 | 解决方案 |
|------|----------|
| LLM 输出 JSON 格式不稳定（被 markdown 代码块包裹） | 增加清洗逻辑 + JSON 解析兜底降级为低置信度 |
| 环境问题误调 LLM 造成成本浪费 | 定义 RULE_BASED_CATEGORIES 集合，环境问题直通不走 LLM |
| AI 生成代码质量不稳定，prompt 软约束常被弱模型无视 | 将质量约束升级为退出码驱动的硬闸门，不依赖模型"自觉" |
| AI 可能整文件覆盖误删已有用例 | precheck 通过 `git show HEAD` 机械比对，检测到删除立即拦截 |
| AI 子智能体循环修复可能无限重试 | loop_gate 读磁盘计数区做机械判定，上下限不依赖对话状态 |
| 上下文压缩后 AI 漏写关键报告段落 | report_check 逐锚点校验 27 个必出区块 |

### 项目成果与效益

- AI 失败自动分类准确率约 **85%**（环境问题 95%+），单次排查时间从 30-60 分钟降至 **3-5 分钟**
- 相似聚合 + 规则直通使 LLM 调用量减少约 40%、额外节省约 30%，显著控制 AI 调用成本
- AI 脚本生成的硬闸门体系将代码规范违规在**零执行成本阶段拦截**，减少阶段三无效重跑
- 知识库已收录多类典型失败模式，命中案例库直接匹配不走 LLM，持续运营迭代中
- 已存在的"已有用例误删"拦截机制堵死了 AI 编码的严重事故风险
