# 基于Codex的测试脚本开发实践

## Codex 工具

得益于LLM和ai coding工具的进化，以及测试框架的用例规模及方法沉淀，无论是开发效率和用户体验上，Codex目前都属于第一梯队级别的存在。

Codex 使用介绍参考 AI Coding工具使用

## 自修复式开发方案

不是“先把代码写完了再跑”，而是采用闭环模式：执行 -> 观测 -> 定位 -> 最小修复 -> 回放验证 -> 继续收敛。

Codex 的价值不只是生成代码，而是把每次运行暴露出的真实信号，转化成下一轮修复输入。

Codex 开发测试脚本，依赖以下输入信号：

- pytest 执行结果
- Playwright 定位报错
- 失败截图
- 页面表格真实数据
- 日志中的步骤轨迹

## 通用要求

先阅读目标文件、同模块现有用例、相关页面对象与 fixture，严格按当前项目风格编写，不要自创一套新写法。

必须遵循以下约束：

- 保持现有 `allure.epic / feature / story / title`、中文 `docstring`、`allure_step_log("步骤N: ...")` 风格
- 优先复用已有 fixture、页面对象方法、工具函数、日志方法、数据加载方式
- 断言必须校验真实业务结果，不能只验证“点击成功”或“页面没报错”
- 断言信息要清楚，尽量包含“期望/实际”
- 用例产生的资源必须显式清理，删除时要使用修改后的最新名称、端口、ID 等
- 如果页面对象缺少能力，只补最小必要实现，不做无关重构
- 不要修改与本次需求无关的逻辑
- 不要随意删除已有 `try/finally`、fixture 回写、teardown、清理步骤，除非需求明确要求
- 写完后必须做最小必要验证，不要只停留在代码修改

## 开发流程要求

你需要按这个顺序完成任务：

1. 先阅读目标测试文件和同模块用例，提炼当前编写风格
2. 再查看相关页面对象、fixture、测试数据是否已有可复用能力
3. 如缺少方法，补充最小可复用页面对象实现
4. 按现有风格补充或修改测试用例
5. 完成代码后，做语法检查
6. 做最小范围 pytest收集或执行验证
7. 最后总结改动文件、验证结果和剩余风险

# 一、页面基本功能验证类用例

适用于：
- 创建、删除、修改、搜索、重置
- 绑定、解绑、启用、停用
- 列表页、详情页、弹窗表单类功能验证

## 编写要求

- 用例结构通常为：准备数据 -> 执行操作 -> 校验结果 -> 清理数据
- 步骤要清晰拆分，通常 3 到 5 步
- 创建类用例要校验列表或详情页字段
- 修改类用例要覆盖需求里所有可修改项，不能漏项
- 搜索类用例要校验搜索结果正确、重置后恢复正常
- 删除类用例要校验资源已从列表消失
- 如 fixture 中资源标识发生变化，需要同步回写 fixture，确保 teardown 正常执行

## 推荐骨架

```python
@allure.title("模块-功能点")
def test_xxx(self, page_fixture, resource_fixture):
    """一句话描述测试目的"""
    xxx = ...

    with allure_step_log("步骤1: 准备测试数据"):
        ...

    with allure_step_log("步骤2: 执行页面操作"):
        ...

    with allure_step_log("步骤3: 验证结果"):
        ...

    with allure_step_log("步骤4: 清理测试数据"):
        ...
```

## 问题和改进

由于需求的多样性，仅靠提示词，无法保证一次能写对测试脚本，AI也依赖更多信息输入（需求文档、页面截图、交互方式、Playwright录制脚本）

对话应一次性、完整地描述开发需求内容，切忌后续对话又加需求

优先跑“1 条最小 happy path”，比如“创建删除xxx”，再扩展到全量

未复用其他层级已存在的fixture
