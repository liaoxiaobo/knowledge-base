### 三、根因分类与推理

#### 3.1 排除项（已排除）

| 排除项 | 排除依据 |
|:---|:---|
| ❌ 用例代码错误 | `evs_create` 调用成功，弹窗提示"创建云硬盘成功" |
| ❌ 断言逻辑错误 | `assert_status` 正确检测状态，只是状态未达预期 |
| ❌ UI 元素定位错误 | Playwright 成功定位到目标表格行（`get_row_by_name` 成功） |
| ❌ 网络/环境问题 | 页面正常加载，其他用例通过，仅大容量共享盘超时 |
| ❌ 存储类型不匹配 | `stor=ceph` 在 `ceph-type` 的 supported_storages 白名单中 |

#### 3.2 根因判定：**环境问题/产品性能问题**

**核心根因**：**Ceph 存储后端创建 500GiB 共享空白云硬盘的实际耗时 > 300 秒，超过了 `assert_status` 的默认超时阈值。**

**推理过程**：

1. **创建操作本身成功**：`evs_create` 完成后，系统弹窗显示"创建云硬盘成功"，说明前端请求已正确提交到后端。

2. **后端状态未收敛**：截图显示断言时刻（约创建后 5 分钟）云硬盘仍处于"创建中"状态，说明 Ceph 后端尚未完成卷的物理创建。

3. **参数特异性**：
   - `params1`（500GiB 非共享镜像盘）通过 → 500GiB 非共享卷可在 300s 内完成
   - `params2`（500GiB **共享**空白盘）失败 → **共享属性 + 空白盘** 的组合在 Ceph 上创建耗时更长
   - 共享卷在 Ceph 上可能涉及额外的元数据操作或同步逻辑

4. **超时阈值不足**：`assert_status` 的默认 300 秒超时对于此场景过于严格。Ceph 在负载较高或卷较大时，创建耗时可能超过 5 分钟。

### 四、可执行的修复建议

#### 方案 A：调整超时时间（推荐，最小改动）

在 `test_volume_create` 中针对 params2 增加 `assert_status` 的 timeout：

```python
# test_evs_create.py:36
# 大容量共享卷可能需要更长时间
timeout = 600 if params.get('shared') and params['size'] >= 500 else 300
evs_page.assert_status(name, status="可用", timeout=timeout)
```

#### 方案 B：在测试数据中标记慢操作

在 `test_data.yaml` 中为大容量共享卷添加标记，配合 fixture 动态调整超时：

```yaml
- empty: true
  size: 500
  shared: true
  slow_create: true  # 标记需要更长等待时间
```

#### 方案 C：环境层面优化

如果 500GiB 共享卷在 Ceph 上稳定需要 >5 分钟，建议：
- 检查 Ceph 集群性能（osd 数量、磁盘类型、网络带宽）
- 检查是否有其他并发操作占用存储资源
- 在 Jenkins 执行时考虑错峰执行大容量用例

### 五、结论

| 维度 | 判定 |
|:---|:---|
| **根因分类** | 🟡 环境问题 / 产品性能边界 |
| **责任归属** | 非用例代码缺陷，是**测试期望（300s 超时）与后端实际性能不匹配** |
| **紧急程度** | 中 — 稳定复现，影响 Jenkins CI 中 evs 模块通过率 |
| **修复成本** | 低 — 单点调整超时参数即可 |
| **是否需要产品介入** | 视情况而定 — 若 Ceph 上 500GiB 共享卷正常创建时间 >5min 是产品设计预期，则测试应适配；若明显慢于正常水平，则产品应排查存储性能 |

> 💡 **建议**：先执行方案 A（增加超时至 600 秒），观察是否可稳定通过。若仍超时，则需产品侧介入分析 Ceph 共享卷创建链路。

| 假设基线 | 实际耗时 | 根因结论 | 分类 |

## 六、核心经验教训

### 6.1 关于"排障报告" vs "结构化结论卡片"

原始输出是一份**排障报告**，大而全，包含推理全过程。
规范要求的是**结构化结论卡片**，精简聚焦。

但规范有一个隐藏陷阱：**为了填满模板，容易过度确定化**。

| 维度 | 排障报告（原始） | 结构化卡片（规范） | 正确做法 |
|:---|:---|:---|:---|
| 信息量 | 大而全 | 精简 | 精简，但**不牺牲诚实性** |
| 结论确定性 | "视情况而定"（模糊但诚实） | "高置信度"（明确但可能过度） | "中置信度+明确说明盲区"（既明确又诚实） |
| 修复建议 | 3个方案 | 1-2条 | 1-2条，且**设计成可验证的实验** |

### 6.2 关于置信度

**"高置信度"是一种很强的承诺。** 只有当：
- 有**直接证据**证明根因
- **所有反例都被排除**
- **没有关键信息盲区**

时才配给"高"。

**本案例的真实状态**：最可能的方向是超时阈值不足，但有一个**无法消除的不确定性**（缺少基线）。所以：
- 主结论：**用例问题**（最可控、最稳妥的方向）
- 置信度：**中**
- 补充：**"无法排除 Ceph 性能异常"**

### 6.3 改进口诀

> 先给元数据（什么时间、什么材料），再给结论（什么分类、多少置信度——**不够高就诚实说中/低**），再给证据（每条支持什么判断），最后给建议（**设计成可验证的实验**）。不要叙事，要结构化卡片；不要为了填满模板而强行确定。

## 七、用户如何约束 AI 自由发挥的空间

### 7.1 当前已有约束手段（以及为什么失效）

| 约束手段 | 形式 | 失效原因 |
|:---|:---|:---|
| **Skill 文档** | Markdown 文本，描述输出格式和流程 | 它是"参考"而非"强制"。AI 可以选择性遵循，也可以声称遵循了但实际变形 |
| **输出约束条款** | "不要输出大而全的报告" | 语言禁令很容易被绕过——写得不那么"大"，但仍然是报告而非卡片 |
| **自检清单** | `[x]` 跟踪 | 依赖 AI 的自觉性。本次实例中完全没有执行 |
| **CLAUDE.md** | 项目级行为准则 | 优先级在 skill 之下，且同样是指南性质 |

**核心问题**：所有这些约束都是**文本层面的建议**，不是**机制层面的强制**。

### 7.2 用户可以增强约束的手段

#### 1. 把 Skill 变成"检查清单"而非"参考指南"

当前的 skill 是：
> "你应该这样输出..."

更有效的方式是：
> "输出前，你必须逐项确认以下检查清单，全部通过后才能输出。任何一项不通过，回溯修正："
> - [ ] 是否有 0. 分析元数据？
> - [ ] 是否有结论（含根因分类+置信度+理由）？
> - [ ] 关键证据是否在 2-4 条之间？
> - [ ] 每条证据是否明确说明了支持什么判断？
> - [ ] ...

这利用了**清单效应**——人在面对逐项打勾时更难跳过。

#### 2. 在对话中显式激活约束

用户可以直接说：
> "请严格按照 `test-failure-analysis` skill 的'输出格式'章节输出。输出前，先向我展示你的自检清单，全部勾选通过后再给出最终报告。"

这比 skill 文档本身更有效，因为**对话指令的权重高于文档引用**。

#### 3. 要求 AI 展示"不符合项"而非直接输出

不直接要结果，而是要先审查：
> "在输出根因分析之前，先列出你的输出与 skill 规范的偏差。确认没有偏差后，再输出报告。"

这迫使 AI 先对照规范，而不是凭直觉写完再对照。

#### 4. 事后反馈建立惩罚机制

每次 AI 偏离规范后，用户的纠正是最有效的约束：
- 指出具体问题（"为什么没有置信度？"）
- 要求重写（"按规范重新输出"）
- 将纠正结果沉淀到案例库（如本文件）

这本质上是在**训练 AI 的上下文记忆**——纠正得越多，AI 在这个项目中越不容易重复犯错。

#### 5. 在 Skill 文档中增加"反例"

当前 skill 只有"应该怎么做"，没有"常见错误示例"。可以增加：

```markdown
## 常见错误输出（禁止）
❌ 错误示例 1：没有置信度，直接给结论
❌ 错误示例 2：证据叙事化，没有说明支持什么判断
❌ 错误示例 3：修复建议超过 2 条
❌ 错误示例 4：把现象当根因（如"超时了"写成根因）
```

**负面示例**比正面规范更有约束力，因为 AI 可以很容易地说"我遵循了正面规范"（变形后），但很难否认自己犯了负面示例中的错误。

### 7.3 最核心的一条

> **约束 AI 最有效的手段，不是让 AI 自我约束，而是让用户在对话中持续施加外部约束。**

Skill 文档是"事前设计"，对话纠正是"事中控制"。前者容易失效，后者几乎一定生效。

本次实例中用户展示了最有效的约束方式：
1. **明确要求**（"帮我找 slow 用例"）
2. **事后审查**（"你反思下输出符合规范吗"）
3. **强制重写**（"按规范重写"）
4. **纠正偏差**（"这样太绝对了，可能会判断错误"）
5. **沉淀经验**（"放到案例库"）

这比任何 skill 文档都管用。如果用户希望 AI 在未来自动遵循规范，最有效的做法是在**每次偏离时都立即纠正**——这比修改 skill 文档更有训练效果。

### 7.4 一个具体的改进建议

可以把 `test-failure-analysis` skill 的"分析流程"章节改成这样：

```markdown
## 分析流程（强制检查清单）

**你必须在输出最终报告前，先向用户展示以下自检清单的勾选结果。**
任何一项未勾选，不得输出报告。

- [ ] 已读取 case_library.md 并判断是否存在匹配模式
- [ ] 已收集所有输入材料（日志、截图、代码）
- [ ] 已确定根因分类（环境/用例/产品 三选一）
- [ ] 已评估置信度（高/中/低）及理由
- [ ] 关键证据在 2-4 条之间，每条说明了支持什么判断
- [ ] 排除项说明了为什么不是另外两类
- [ ] 修复建议不超过 2 条，且可直接执行

**展示格式：**
### 自检结果
- [x] 已读取案例库 → 匹配/不匹配模式 XXX
- [x] 已收集材料 → 日志/截图/代码
...

**只有在用户确认自检通过后，你才能输出最终的根因报告。**
```

这样用户就有了**事中控制点**——AI 必须在输出前停下来给用户看清单，用户可以否决或要求修正。
