# AI智能失败归因 Skill 实战

> **摘要**：在 sugoncloud/playwright-sugon 项目中，基于 Claude Code Skill 实现 Playwright+Pytest+Allure 测试失败的自动化根因分析，把"人工逐条查看 Allure 报告"的排查模式转为"AI 先定位、人工再确认"的人机协同模式，缩短失败定位时间并沉淀可复用的失败模式库。

## 三、测试场景设计

本 Skill 面向"测试失败后需要根因分析"的场景。以下为主要验证场景：

| 场景                | 输入                            | 预期                  | 实际结果      | 状态  |
| ----------------- | ----------------------------- | ------------------- | --------- | --- |
| 状态断言超时            | result.json + 截图显示目标资源状态正常    | 根因：用例问题（动态表格行定位漂移）  | 实际输出符合预期  | ✅   |
| 弹窗/表单操作后断言超时      | result.json + 截图显示弹窗未关闭、输入框为空 | 根因：用例问题（fill 静默失效）  | 实际输出符合预期  | ✅   |
| 进入详情页后找不到数据行      | result.json + 截图显示表格区域未渲染     | 根因：用例问题（异步数据未等待）    | 实际输出符合预期  | ✅   |
| 大容量共享卷状态长时间为"创建中" | result.json + 截图显示中间态         | 根因：环境问题（后端状态收敛超时）   | 实际输出符合预期  | ✅   |
| 批量删除时确认对话框未找到     | result.json + 日志显示勾选成功但对话框未弹出 | 根因：用例问题（轮询清空表格勾选状态） | 实际输出符合预期  | ✅   |
| 列表页搜索后找不到目标       | result.json + 搜索前后截图          | 需先验证搜索功能是否生效，再判定根因  | 已加入强制排查步骤 | ✅   |

## 四、遇到的问题与复盘

### 问题 1：AI 容易把"超时"直接当根因

- **现象**：早期版本输出"断言超时失败"
- **根因**：把报错表象当成了根因，没有追问"为什么超时"
- **解决**：在 SKILL.md 中强制要求"区分报错表象和真实根因"，并加入交叉验证步骤
- **沉淀**：纳入 Skill 输出格式自检清单

### 问题 2：缺少 trace 时高置信度误判

- **现象**：AI 在没有 trace 的情况下给出"高置信度"结论
- **根因**：模型基于代码推断过度自信
- **解决**：加入材料覆盖度与置信度映射规则，缺少 trace 时最高给"中置信度"
- **沉淀**：写入 SKILL.md 的"输出前自检"和"材料覆盖度与置信度映射"章节

### 问题 3：搜索/过滤失败被误判为"目标不存在"

- **现象**：列表页搜索后找不到目标，AI 推断目标不存在
- **根因**：未验证搜索功能本身是否生效
- **解决**：在 SKILL.md 中加入强制排查步骤 —— 先对比搜索前后结果集变化，再下结论
- **沉淀**：形成"搜索/过滤类失败"专项排查 Checklist
