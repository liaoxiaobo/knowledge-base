# AI 编程工具选型与 Prompt 实践

> 整理日期：2026-06-16  
> 适用场景：选择适合自己的 AI 编程工具、编写高效 Prompt、建立个人 AI 辅助测试工作流

---

## 一、主流 AI 编程工具对比

| 工具 | 核心优势 | 适用场景 | 局限 |
|---|---|---|---|
| **Claude Code** | 上下文理解强、终端集成深、工具链完整 | 复杂代码库重构、多文件修改、测试脚本生成 | 需要一定学习成本 |
| **GitHub Copilot** | IDE 集成好、补全流畅 | 日常编码补全、单函数生成 | 复杂上下文理解有限 |
| **Cursor** | 编辑器体验好、Composer 模式适合多文件 | 中小型项目开发、快速原型 | 大项目上下文管理能力弱于 Claude Code |
| **ChatGPT / Claude Web** | 通用问答、长文本理解 | 方案设计、知识问答、文档生成 | 不直接操作代码库 |
| **DeepSeek / 通义千问** | 中文理解好、成本低 | 日志分析、中文文档生成 | 代码生成稳定性待验证 |

**建议**：Claude Code 作为主力工具负责代码库级任务，Copilot/Cursor 作为 IDE 内补全辅助，Web 版大模型用于方案设计和知识查询。

---

## 二、工具选型决策树

```text
是否需要直接操作代码库？
  ├── 是 → 上下文复杂、多文件 → Claude Code
  │        简单补全、单文件 → Copilot / Cursor
  └── 否 → 需要长文本/方案设计 → Claude / ChatGPT Web
            需要中文处理/低成本 → DeepSeek / 通义千问
```

## 三、Prompt 编写原则

### 3.1 明确目标

❌ 不佳："帮我写测试"  
✅ 更好："为 `/api/orders` POST 接口生成 pytest 用例，覆盖正常下单、库存不足、未登录三种场景，使用现有的 `client` fixture。"

### 3.2 提供上下文

- 引用具体文件路径。
- 说明项目使用的测试框架（pytest / unittest / Robot Framework）。
- 给出相关代码示例或接口契约。

### 3.3 指定输出格式

```text
"请按以下格式输出：
1. 测试函数名
2. 前置条件
3. 测试步骤
4. 断言点
5. 需要的 fixture"
```

### 3.4 给出约束条件

- 不要修改现有公共函数。
- 保持与现有用例风格一致。
- 使用参数化减少重复代码。

### 3.5 要求可验证

```text
"生成后请说明如何运行这些用例，以及预期通过条件。"
```

## 四、测试开发高频 Prompt 模板

### 4.1 生成接口测试用例

```text
请为以下接口生成 pytest 测试用例：

接口：POST /api/users
请求体：{ "name": string, "email": string, "role": string }
响应：201 创建成功，返回 { "id": int, "name": string, "email": string }

要求：
1. 正常场景 1 条
2. 参数校验异常场景至少 3 条（邮箱格式、角色非法、姓名为空）
3. 使用 requests 和已有的 base_url fixture
4. 每个用例有清晰的 Arrange-Act-Assert 三段
```

### 4.2 分析测试失败

```text
以下 pytest 用例失败，请分析可能原因并给出修复建议：

[粘贴 traceback]

相关代码：
[粘贴测试代码和被测代码]
```

### 4.3 重构重复代码

```text
请识别 tests/ 目录下重复的请求构造代码，并提取到公共辅助函数或 fixture 中。
要求：
1. 不改变现有用例行为
2. 减少代码重复
3. 保持可读性
```

### 4.4 生成测试数据

```text
请生成 100 条符合以下 schema 的测试数据，要求覆盖正常值、边界值、异常值：

{
  "order_id": string(20),
  "amount": float(0.01~999999.99),
  "status": enum("pending", "paid", "shipped", "cancelled"),
  "create_time": ISO8601 timestamp
}

输出为 CSV 格式，并说明异常值的选取理由。
```

### 4.5 编写测试方案

```text
请为 [某功能] 编写测试方案，包含：
1. 测试目标与范围
2. 测试策略（功能/接口/性能/兼容性）
3. 测试环境要求
4. 用例设计思路
5. 风险与准入准出标准
```

## 五、建立个人 AI 工作流

```text
1. 需求理解：用 AI 梳理测试点和边界条件
2. 用例设计：让 AI 生成初稿，人工补充异常场景
3. 代码实现：AI 生成骨架，人工调整断言和 fixture
4. 失败分析：AI 辅助定位，人工确认根因
5. 文档沉淀：把最终方案写回知识库
```

---

## 六、实践建议

1. **不要完全依赖 AI**：AI 生成的是草稿，关键断言和边界条件必须人工把关。
2. **建立 Prompt 库**：把验证有效的高频 Prompt 保存到 `8.06 AI Prompt 模板库`。
3. **迭代优于一次性**：先让 AI 做小部分，验证方向正确后再扩大范围。
4. **保护敏感信息**：生产数据、内网地址、密钥不要输入给 AI。
5. **持续调优 Prompt**：记录哪些 Prompt 效果好，逐步沉淀为 Skills。
