## 一、为什么测试需要专门的数据构造策略

- **用例隔离**：避免一个用例的数据影响另一个用例。
- **环境可重复**：每次测试都能在一致的基线数据上运行。
- **边界覆盖**：需要构造空值、超长、特殊字符、并发冲突等异常数据。
- **效率要求**：大量数据时不能靠手工插入。

## 二、测试数据构造策略

### 2.1 基线数据 + 增量数据

| 策略 | 说明 | 适用场景 |
|---|---|---|
| 基线数据 | 测试开始前初始化一套标准数据 | 回归测试、冒烟测试 |
| 增量数据 | 每个用例自行创建并清理 | 单元测试、集成测试 |
| 快照恢复 | 测试前后恢复数据库快照 | 复杂业务链路测试 |

### 2.3 工厂模式生成数据

```python
import random
import string

def create_user(overrides=None):
    defaults = {
        "username": f"user_{random_string(6)}",
        "email": f"{random_string(6)}@test.com",
        "status": 1,
    }
    if overrides:
        defaults.update(overrides)
    return defaults

def random_string(length=8):
    return "".join(random.choices(string.ascii_lowercase, k=length))
```

### 2.4 使用 Faker 生成更真实的数据

from faker import Faker

fake = Faker("zh_CN")

### 2.5 数据脱敏

从生产环境导出的数据用于测试前，必须脱敏：

```python
import re

def mask_phone(phone):
    return re.sub(r"(\d{3})\d{4}(\d{4})", r"\1****\2", phone)

def mask_id_card(id_card):
    return id_card[:6] + "********" + id_card[-4:]
```

## 三、数据库断言

### 3.1 断言内容

| 断言类型 | 示例 |
|---|---|
| 记录存在性 | 创建用户后，数据库中是否存在该用户 |
| 字段值 | 订单状态是否为 `paid` |
| 记录数 | 某条件查询结果是否为预期数量 |
| 关联数据 | 订单与用户、商品关联是否正确 |
| 时间字段 | `created_at` 是否在合理范围内 |

### 3.2 Python 示例

```python
import pymysql

def assert_user_exists(db, user_id, expected_status):
    with db.cursor() as cursor:
        cursor.execute(
            "SELECT status FROM users WHERE id = %s", (user_id,)
        )
        row = cursor.fetchone()
        assert row is not None, f"用户 {user_id} 不存在"
        assert row["status"] == expected_status, f"状态期望 {expected_status}，实际 {row['status']}"
```

### 3.3 使用 SQLAlchemy 断言

```python
from myapp.models import Order

def assert_order_amount(session, order_no, expected_amount):
    order = session.query(Order).filter_by(order_no=order_no).first()
    assert order is not None
    assert order.amount == expected_amount
```

## 四、数据隔离与清理

### 4.1 事务回滚

每个测试用例在独立事务中执行，结束后回滚，不污染数据库。

```python
import pytest

@pytest.fixture
def db_session():
    session = Session()
    yield session
    session.rollback()
    session.close()
```

### 4.2 setup/teardown 清理

```python
@pytest.fixture
def test_user(db):
    user_id = create_user(db, "test_user")
    yield user_id
    delete_user(db, user_id)  # 清理
```

### 4.3 数据库快照

对于复杂测试，可在测试前导出快照，测试后恢复。

```bash
# 测试前备份
mysqldump -u root -p test_db > baseline.sql

# 测试后恢复
mysql -u root -p test_db < baseline.sql
```

## 六、实践建议

1. **每个用例自包含**：创建自己的数据并清理，不依赖其他用例的执行顺序。
2. **避免共享全局测试数据**：共享数据容易导致用例耦合、误报。
3. **时间字段用相对时间**：避免硬编码 `2020-01-01` 导致过期失败。
4. **大数据量测试单独管理**：构造百万级数据时使用批量插入、临时表。
5. **敏感数据绝不直接复用生产库**：必须脱敏或合成。
